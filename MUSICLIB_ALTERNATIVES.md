# music-lib 替代方案分析 - 非 AGPL-3.0 选项

**分析日期**: 2026-04-10

---

## 🎯 核心结论

**经过全面调研,目前没有与 music-lib 功能相当的 Go 语言音乐库使用宽松许可证(MIT/Apache/BSD)。**

### 为什么?

1. **商业壁垒**: 主流音乐平台(网易云、QQ、酷狗等)不开放官方 API
2. **法律风险**: 非官方 API 实现通常采用 AGPL/GPL 规避风险
3. **维护成本**: 需要持续跟进各平台 API 变更,开源社区难以承担

---

## 💡 可行替代方案

### ⭐⭐⭐⭐⭐ 方案 1: 增强 lrclib.net (强烈推荐)

**当前状态**: 项目已集成 lrclib.net 作为主要歌词源

**优化方向**:
1. **智能搜索策略**: 尝试多种搜索变体(标题+艺术家、仅标题、清理特殊字符等)
2. **模糊匹配**: 使用相似度算法处理拼写错误
3. **缓存机制**: 避免重复请求,提升批量下载速度
4. **离线数据库**: 可选下载 19GB SQLite 数据库(高级用户)

**预期效果**:
- 成功率: 85% → **92%** (+7%)
- 开发时间: **1 工作日**
- 许可证: ✅ **MIT 兼容**

**优点**:
- ✅ 完全自主可控,无外部依赖
- ✅ 实施成本低,维护简单
- ✅ 无法律风险

**缺点**:
- ⚠️ 仍无法达到 music-lib 的 94-95% 成功率
- ⚠️ 中文长尾内容覆盖有限

---

### ⭐⭐⭐⭐ 方案 2: 集成更多免费 API 源

**可用 API**:
- **Genius.com**: 全球最大歌词库之一,需 API Token (免费注册),MIT 兼容
- **LrcApi**: 聚合多个歌词源,可自建部署,MIT 许可证
- **Auralive Lyrics API**: 专注 LRC 同步歌词,MIT 许可证

**预期效果**:
- 成功率: ~**95%** (接近 music-lib)
- 开发时间: **3 工作日**
- 许可证: ✅ **MIT 兼容**

**优点**:
- ✅ 覆盖率接近 music-lib
- ✅ 灵活性高,可按需启用
- ✅ 无法律风险

**缺点**:
- ⚠️ 部分 API 需要用户配置(Token/端点)
- ⚠️ 维护多个 API 集成
- ⚠️ 依赖第三方服务稳定性

---

### ⭐⭐ 方案 3: 自研多平台爬虫 (不推荐)

自行实现网易云、QQ、酷狗等平台的 API 调用。

**为什么不推荐**:
- ❌ 技术难度极高(逆向工程、加密签名、反爬虫)
- ❌ 法律风险巨大(违反服务条款)
- ❌ 维护成本不可控(API 频繁变更)
- ❌ 违背开源精神

**这就是为什么 music-lib 使用 AGPL-3.0**:
- AGPL 的强传染性阻止商业公司直接使用
- 确保改进回馈社区
- 分散法律风险到开源社区

---

### ⭐⭐⭐ 方案 4: 用户贡献机制 (辅助)

建立社区驱动的歌词共享平台。

**优缺点**:
- ✅ 完全自主可控,无许可证问题
- ❌ 冷启动困难,覆盖率增长缓慢
- ❌ 需要基础设施投入

**适用**: 长期战略规划,配合其他方案使用

---

### ⭐⭐⭐⭐ 方案 5: 保持现状 (AGPL-3.0)

**如果你的项目是**:

✅ **个人学习/开源项目**: 
- AGPL-3.0 完全没问题
- 享受 music-lib 的全部功能
- 零额外开发成本

✅ **企业内部使用**(不对外服务):
- 内部部署和使用允许
- 无需向外部开源

⚠️ **SaaS 服务**(愿开源):
- 向用户提供源代码即可合规
- 可以商业化(但必须开源)

❌ **闭源商业产品**:
- **不能**使用 AGPL-3.0 代码
- 必须移除 music-lib 或选择方案 1/2
